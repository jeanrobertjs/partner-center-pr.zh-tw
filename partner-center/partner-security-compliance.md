><span data-ttu-id="c2173-179">對於已使用 Azure AD 使用者保護基準原則來實作 MFA 的合作夥伴，請務必注意，使用者保護是以風險為基礎的原則。</span><span class="sxs-lookup"><span data-stu-id="c2173-179">For partners who have implemented MFA using Azure AD End-User Protection baseline policy, it is important to note that End-User Protection is a risk-based policy.</span></span> <span data-ttu-id="c2173-180">此原則所涵蓋的使用者只有在進行有風險的登入嘗試 (例如，使用者從其他位置登入) 時，才會收到要進行 MFA 的提示。</span><span class="sxs-lookup"><span data-stu-id="c2173-180">Users covered by the policy will be prompted for MFA only during risky sign-in attempts (e.g., user is signing in from a different location).</span></span> <span data-ttu-id="c2173-181">此外，此原則下的使用者最多有 14 天的時間來註冊 MFA。</span><span class="sxs-lookup"><span data-stu-id="c2173-181">In addition, users under the policy have up to 14 days to register for MFA.</span></span> <span data-ttu-id="c2173-182">未完成 MFA 註冊的使用者在 14 天的期間內，不會有 MFA 驗證的挑戰。</span><span class="sxs-lookup"><span data-stu-id="c2173-182">Users who haven't complete MFA registration will not be challenged for MFA verification during the 14-day period.</span></span> <span data-ttu-id="c2173-183">因此，使用 Azure AD 使用者保護基準原則來實作 MFA 的合作夥伴，計量可能不會是 100%。</span><span class="sxs-lookup"><span data-stu-id="c2173-183">Therefore, it is expected that the metrics may not be 100% for partners who have implemented MFA using Azure AD End-User Protection baseline policy.</span></span>

### <a name="are-you-using-3rd-party-mfa-solution"></a><span data-ttu-id="c2173-184">您是否使用第三方 MFA 解決方案？</span><span class="sxs-lookup"><span data-stu-id="c2173-184">Are you using 3rd party MFA solution?</span></span>

<span data-ttu-id="c2173-185">如果您使用第三方 MFA 解決方案，請識別您要將它與 Azure AD 整合的方式。</span><span class="sxs-lookup"><span data-stu-id="c2173-185">If you are using 3rd party MFA solution, identify how you are integrating it with Azure AD.</span></span> <span data-ttu-id="c2173-186">一般來說有兩種方法，包括同盟和自訂控制項：</span><span class="sxs-lookup"><span data-stu-id="c2173-186">In general, there are two methods, including federation and custom controls:</span></span>

* <span data-ttu-id="c2173-187">**身分識別同盟**  – 當 Azure AD 收到驗證要求時，Azure AD 會將使用者重新導向至同盟識別提供者進行驗證。</span><span class="sxs-lookup"><span data-stu-id="c2173-187">**Identity federation** – When Azure AD receives an authentication request, Azure AD will redirect the user to the federated identity provider for authentication.</span></span> <span data-ttu-id="c2173-188">驗證成功之後，同盟識別提供者會連同 SAML 權杖，將使用者重新導向回 Azure AD。</span><span class="sxs-lookup"><span data-stu-id="c2173-188">Upon successful authentication, the federated identity provider will redirect the user back to Azure AD along with a SAML token.</span></span> <span data-ttu-id="c2173-189">為了讓 Azure AD 在向同盟識別提供者進行驗證時，辨識使用者已完成 MFA 驗證，SAML 權杖必須包含authenticationmethodsreferences  宣告 (具有 multipleauthn  值)。</span><span class="sxs-lookup"><span data-stu-id="c2173-189">In order for Azure AD to recognize that the user has completed MFA verification when authenticating to the federated identity provider, the SAML token must include the *authenticationmethodsreferences* claim (with value *multipleauthn*).</span></span> <span data-ttu-id="c2173-190">檢查同盟識別提供者是否支援發出這類宣告。</span><span class="sxs-lookup"><span data-stu-id="c2173-190">Check whether that the federated identity provider supports issuing such a claim.</span></span> <span data-ttu-id="c2173-191">若是如此，請檢查同盟識別提供者是否已設定為執行此動作。</span><span class="sxs-lookup"><span data-stu-id="c2173-191">If so, check whether the federated identity provider has been configured to do so.</span></span> <span data-ttu-id="c2173-192">如果宣告遺失，Azure AD (進而是合作夥伴中心) 將不會知道使用者已完成 MFA 驗證，而這會造成計量不是 100%。</span><span class="sxs-lookup"><span data-stu-id="c2173-192">If the claim is missing, Azure AD (and therefore Partner Center) will not know that the user has completed MFA verification and this can cause the metric to not be 100%.</span></span>

* <span data-ttu-id="c2173-193">**自訂控制項** – Azure AD 自訂控制項不能用來識別使用者是否已透過第三方 MFA 解決方案完成 MFA 驗證。</span><span class="sxs-lookup"><span data-stu-id="c2173-193">**Custom Control** – Azure AD Custom Control cannot be used to identify whether a user has completed MFA verification through a 3rd party MFA solution.</span></span> <span data-ttu-id="c2173-194">因此，透過自訂控制項完成 MFA 驗證的任何使用者，一律會對 Azure AD (接著是合作夥伴中心) 顯示未完成 MFA 驗證。</span><span class="sxs-lookup"><span data-stu-id="c2173-194">As a result, any user who has completed MFA verification through a custom control will always appear to Azure AD (and in turn Partner Center) as not having completed MFA verification.</span></span> <span data-ttu-id="c2173-195">可能的話，建議您在與 Azure AD 整合時，切換為使用身分識別同盟，而不是自訂控制項。</span><span class="sxs-lookup"><span data-stu-id="c2173-195">Where possible, it is recommended that you switch to using Identity Federation as opposed to Custom Control when integrating with Azure AD.</span></span>
